# Предфинальная точка 2026-05-11

Этот документ фиксирует рабочую линию, от которой дальше отталкиваемся.
Главное правило: чистая базовая прошивка заливается через ST-Link только для
восстановления платы, а все рабочие обновления ставятся штатно через USB update
режим автора.

## Режимы

### Mode 1: штатный canbox + sideband

Файл:

`firmware/variants_20260511/07_v08_mode1_canbox_cdc_can_uart_USB.bin`

Основа:

- прошивка автора `37FFDA054247303859412243_04350008.bin`;
- UID адаптера `37 FF DA 05 42 47 30 38 59 41 22 43`;
- версия пакета `04 35 00 08`;
- USB identity не меняется, это тот же штатный CDC/proprietary canbox протокол.

Поведение:

- обычный canbox стартует сразу после питания;
- штатные функции автора v08 остаются в `mode1`;
- приложение может читать CAN и UART через наши USB sideband-команды;
- отдельного `gs_usb` в рабочем пути нет, чтобы не менять USB-интерфейс.

Sideband-команды:

| Команда | Назначение |
|---|---|
| `0x70` | CAN log start/stop, копии кадров `C-CAN` и `M-CAN` через штатный USB |
| `0x71` | UART RX mirror status |
| `0x72` | UART RX mirror read |
| `0x73` | UART raw TX через USART2 `PA2` |
| `0x74` | Raw CAN TX в `CAN1/C-CAN` или `CAN2/M-CAN` |

Проверка после заливки:

- USB порт поднялся как `/dev/cu.usbmodemKIA1`;
- `0x70` отвечает `bb a1 41 07 70 01 15`;
- `0x71` отвечает `UART`, `available=0`, `dropped=0`;
- мусорный счетчик UART исправлен первичной инициализацией RAM.

### Mode 2: штатный update

Переход выполняется штатной USB-командой `0x55`. Это тот же механизм, который
использует APK автора: адаптер сам переподключает USB, принимает блоки прошивки
и после завершения возвращается в обычную работу.

Mode2 нужен для:

- установки новых обновлений автора;
- установки наших USB update пакетов;
- восстановления предфинального файла без ST-Link, если штатный update еще
работает.

## Что это дает приложению

Приложение должно работать с одним USB CDC устройством и не переключать адаптер
в отдельный `gs_usb` режим.

Ожидаемый поток:

```text
mode1 штатный canbox работает как раньше
        |
        +-- sideband CAN RX: C-CAN 500k + M-CAN 100k
        +-- sideband UART RX mirror: то, что получает canbox от TEYES/Raise
        +-- sideband UART TX: тестовые/подтвержденные команды в TEYES/Raise
        +-- sideband CAN TX: тестовые/подтвержденные кадры в C-CAN/M-CAN
```

Для интерфейса это значит:

- слева можно показывать UART TX, то есть что приложение/адаптер отправляет в
  магнитолу;
- справа можно показывать UART RX, то есть что приходит от магнитолы или
  внешнего Raise/Simple Soft canbox;
- CAN отдельно разделяется на `M-CAN ch0` и `C-CAN ch1`;
- OBD-подобный экран берет скорость, обороты, температуры, АКБ и другие
  показатели из CAN sideband, а не из классического ELM327 OBD.

## Ограничения

- Это не стандартный OBD/ELM327 и не стандартный SocketCAN/gs_usb.
- Чтение CAN в приложении работает только если приложение реализует наши
  команды `0x70..0x74`.
- `mode1` не надо патчить через ST-Link. ST-Link использовать только для
  восстановления чистой базы.
- Clean-C прошивка в `firmware/custom_c/` остается исследовательской веткой.

## Дальнейшая разработка

1. Довести Android-приложение до работы с sideband `0x70..0x74`.
2. Разделить экспорт на `M-CAN`, `C-CAN`, `UART RX`, `UART TX`,
   `CAN↔UART learned table`.
3. Закреплять подтвержденные состояния: двери, задний ход, климат, температуры,
   обороты, скорость, АКБ.
4. Отдельно добить медиа/FM/навигацию/компас по логике автора и live UART.
5. После накопления полной таблицы уже переносить подтвержденную логику в
   собственную clean-C прошивку.
