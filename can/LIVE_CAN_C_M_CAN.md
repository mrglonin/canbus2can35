# Live CAN-C / M-CAN Map

Короткая карта по live-захватам Sportage/TEYES/2CAN35.

## Шины

- `bus=0` - CAN-C, основной поток двигателя/кузова.
- `bus=1` - M-CAN, медленный поток мультимедиа/сервисных статусов.
- Адаптер: `/dev/cu.usbmodemKIA1`.
- Захват: `python3 repo/tools/stockusb_canlog_2can35.py /dev/cu.usbmodemKIA1 --poll-raw --seconds N`.

## CAN-C / bus0

| ID | Данные | Статус | Примечание |
| --- | --- | --- | --- |
| `0x316` | `DATA[2..3]`, `DATA[6]` | confirmed | RPM и скорость. RPM = `(DATA[2] + DATA[3] * 256) / 4`, speed = `DATA[6]` km/h. |
| `0x329` | `DATA[1]` | confirmed | Температура ОЖ: `(DATA[1] - 0x40) * 0.75`. |
| `0x044` | `DATA[3]` | confirmed | Наружная температура: `(DATA[3] - 0x52) / 2`. |
| `0x541` | bits | confirmed | Кузовные статусы: двери/капот/багажник по `confirmed_can_signals.csv`. |
| `0x553` | bits | confirmed | Задние двери по `confirmed_can_signals.csv`. |
| `0x2B0` | меняется постоянно | candidate | Динамика/руль/движение, точная раскладка не закреплена. |
| `0x4F4` | full payload / bit masks | confirmed live | Парктроники/заднее препятствие, не подтвержденный RCTA. |
| `0x545` | `DATA[3] * 0.1 V` | historical | Был в V18 snapshot как напряжение, но в текущем live-потоке не пришел. |

### Parking Sensors / Rear Obstacle: `0x4F4`

Обычное состояние:

```text
bus=0 id=0x4F4 data=0000C00000000001
```

Активное событие начинается с `0001...`. По сверке с Raise/RZC parking
protocol это похоже на packed-зоны/дистанции парктроников, а не на RCTA.

Пойманные левые события:

```text
0001C00000003001
0001C00100080409
```

Пойманные правые события:

```text
0001C00002000805
0001C00010000841
0001C00018000C61
0001C01218100C61
0001C01318180C19
0001C01303180C19
```

Хвост/сброс после события часто:

```text
0001C00000000001
0001C00003000C07
```

Практическая логика:

- `0x4F4 == 0000C00000000001` - нет активного препятствия/idle.
- `0x4F4` начинается с `0001` - активны parking-зоны.
- Лево/право из этих payload нельзя считать подтвержденным RCTA.
- Дополнительные отличия в payload похожи на дальность/силу/зону приближения.

## M-CAN / bus1

| ID | Данные | Статус | Примечание |
| --- | --- | --- | --- |
| `0x131` | `1000FF001000FF00` | observed stable | Стабильный 1 Hz статус. |
| `0x132` | `9001011400000001` | voltage candidate | Хороший кандидат напряжения сети: `DATA[0] / 10 = 14.4 V`. |
| `0x134` | `0300000100001000` | observed stable | Стабильный 1 Hz статус. |
| `0x169` | например `200103FF14004000`, `270103FF14004000` | observed | Статус режима/передачи/движения; менялся в live-захватах, но blind spot не основной здесь. |
| `0x497` / `0x4F7` | `300005AAAAAAAAAA` | transient | Редкие ISO-TP Flow Control кандидаты, для blind spot не закреплены. |

### Voltage

Текущий рабочий источник напряжения - M-CAN `0x132`:

```text
bus=1 id=0x132 data=9001011400000001
DATA[0] = 0x90 = 144
voltage = DATA[0] / 10 = 14.4 V
```

`0x132` был стабильным в live-захватах. Стандартный `0x545` из старого snapshot в текущем потоке не появился.

## Diagnostics

Пробовали диагностические запросы:

- functional OBD `0x7DF`: `01 00`, `01 0C`, `01 0D`, `01 05`, `01 42`, `09 02`, `22 F190`;
- physical `0x7E0/0x7E1`;
- UDS probe `0x700..0x74F` с `10 01`;
- 29-bit `0x18DB33F1`, `0x18DAxxF1`.

Адаптер отвечал `ACK 00` на передачу, но ответных CAN-кадров `0x7E8..0x7EF` / `0x700..0x7FF` не было. На этом tap надежнее использовать passive CAN.
